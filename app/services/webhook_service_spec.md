# Спецификация: Сервис вебхуков (WebhookService)

> **Файл реализации:** `webhook_service.py`  
> **Слой:** Services / Use Cases  
> **Ответственность:** Обработка входящих webhook-отчётов от провайдеров (Режим Б — Пассивный слухач)

---

## 1. Общие правила

- Сервис обрабатывает входящие JSON-отчёты об инцидентах безопасности (Guardrail).
- Каждый отчёт привязывается к исходному запросу через `trace_id`.
- Сервис **не знает** о HTTP/FastAPI — принимает чистые данные.
- Валидация токена безопасности (`X-Webhook-Secret`) — ответственность слоя `api/`.

---

## 3. Ограничения входящего payload (защита от DoS)

Для предотвращения атак типа «отказ в обслуживании» через сверхбольшие или глубоко вложенные JSON-тела, на уровне роута (webhook.py) должны быть реализованы следующие проверки **до** передачи данных в сервис:

- **Максимальный размер тела запроса:** 1 мегабайт. Если заголовок Content-Length превышает лимит или фактический размер тела больше лимита — немедленно вернуть HTTP 413 Payload Too Large.
- **Максимальная глубина вложенности JSON:** 10 уровней. При обнаружении более глубокой вложенности — вернуть HTTP 422 с описанием ограничения.
- Эти проверки выполняются на уровне API-слоя, сервис получает уже валидированные данные.

## 4. Метод: process_guardrail_incident

### Назначение

Обработка входящего webhook-отчёта о срабатывании Guardrail.

### Описание интерфейса

- **Асинхронный метод**, принимающий один параметр: `payload` (словарь — тело входящего webhook-запроса).
- **Возвращает:** словарь с подтверждением обработки.

### Пошаговая логика

1. **Извлечение trace_id из payload:**
   - Искать `trace_id` в корне payload.
   - Если `trace_id` отсутствует или пуст → искать в `payload.get("metadata", {}).get("trace_id")`.
   - Если всё равно не найден → сгенерировать новый UUID и пометить как `"trace_id_source": "generated"`.

2. **Валидация формата trace_id:**
   - Проверить, что `trace_id` соответствует формату UUID v4.
   - Если невалидный формат → использовать как есть, но добавить предупреждение в лог.

3. **Проверка связи с исходным запросом:**
   - Вызвать `log_repo.get_by_trace_id(trace_id)`.
   - Если найдены записи с `event_type="chat_request"` → инцидент **связан** с конкретным промптом.
   - Если не найдены → инцидент **осиротевший** (orphaned), но всё равно записывается.

4. **Формирование записи инцидента** — словарь со следующими ключами:
   - `original_webhook_body` — исходное тело webhook-запроса (из параметра `payload`)
   - `trace_id_source` — источник trace_id: `"webhook"` (если извлечён из payload) или `"generated"` (если сгенерирован системой)
   - `linked_to_prompt` — булево значение: связан ли инцидент с исходным промптом
   - `processed_at` — текущая дата-время в формате ISO 8601 (UTC)

5. **Запись в журнал:**
   - Вызвать `log_service.log_guardrail_incident(trace_id, incident_payload)`.

6. **Возврат подтверждения** — словарь со следующими ключами:
   - `status` — значение `"accepted"`
   - `trace_id` — сквозной идентификатор (использованный или сгенерированный)
   - `linked_to_prompt` — булево значение: связан ли инцидент с исходным промптом

---

## 5. Обработка ошибок

| Ситуация                        | Действие                                          |
|---------------------------------|---------------------------------------------------|
| Пустой payload                  | Вернуть словарь со статусом "rejected" и причиной "empty payload" |
| Отсутствует trace_id            | Сгенерировать новый, записать с пометкой          |
| Ошибка записи в БД             | Логировать ошибку, вернуть словарь со статусом "error" |
| Невалидный JSON в payload       | Обрабатывается на уровне api/ (Pydantic)          |
